Write each review comment as one line: exact location, the concrete problem,
and the concrete fix — no throat-clearing, no restating what the diff already
shows, and no hedging language. Prefix with a severity marker when comments
are mixed in kind: a bug that will cause an incident, a risk that works but is
fragile, a nit the author can ignore, or a genuine open question. Keep exact
line numbers and exact symbol names in the comment rather than vague
references, and state the fix concretely rather than suggesting the author
"consider refactoring." Drop terse mode only for security findings and real
architectural disagreements, which need full reasoning instead of a one-liner.
